## Novelty: auto-rolling on-chain market discovery

The strategy bot has **no hardcoded PT market address**. Every epoch it:

1. Calls `listAllMarkets(exponentProgram)` → returns all 94 `MarketTwo` accounts
2. Filters for ONyc-backed SY mint whitelist (maturity + liquidity + APY bounds)
3. Ranks by score = maturity-proximity + APY + log10(depth)
4. Picks the best live candidate
5. If held PT is matured (next epoch), redeems first, then deploys into the new market

Verified live on mainnet:

```
[scan]  94 MarketTwo accounts fetched in 1380ms
[cat]   3 ONyc-backed markets: 2 matured, 1 live
[pick]  ✅ 8QJRc12BDXHR... — 29.6d, 13.87% APY, $3.69B SY depth
[+30d]  no live → auto-fallback to 100% Kamino ✅
```

Code location: `strategy-bot/src/adapters-sdk/exponent.ts` → `findBestOnycMarket()`.

## Honest caveats

<AccordionGroup>
  <Accordion title="Points are a promise, not a token">
    $SAVE points follow the EigenLayer / Pendle / Ethena pattern. Recorded on-chain deterministically; redemption is a stated intention, not a contractual commitment.
  </Accordion>
  <Accordion title="10.42% APY uses live numbers, but rates move">
    Kamino APR floats and PT rates move daily. The claim is trailing, not a guaranteed floor. The [Yield Overview](/yield/overview) quotes the 10.3% design target; [the reinsurance detail page](/yield/reinsurance) explains the fixed-sleeve mechanics.
  </Accordion>
  <Accordion title="OnRe has 8-month track record">
    Mitigated by 52-day PT tenor + auto-roll. See the [Safety FAQ](/faq/safety) for detailed counterparty disclosure.
  </Accordion>
  <Accordion title="PT-ONyc pool depth ~$6.9M">
    $100k deploys cleanly. A $1M seed would need laddering — the auto-roll infrastructure supports this trivially via multiple markets.
  </Accordion>
  <Accordion title="Bot keypair = admin keypair (hackathon scope)">
    Production separation deferred. Cobo MPC infra is the planned path (sponsor prize in this hackathon).
  </Accordion>
  <Accordion title="Rolling jackpot rent accounting">
    `prize_pool_balance` inflates virtually; actual payout is capped by real token balance. Documented in `settle_draw.rs`.
  </Accordion>
